# Atlas Finance

Atlas Finance — личное PWA-приложение для управления денежным потоком между выплатами. Цель проекта — не бухгалтерский учет ради учета, а быстрый ответ на практический вопрос: сколько денег можно безопасно потратить сегодня, чтобы не уйти в кассовый разрыв и начать копить.

## Контекст

Пользователь сейчас ведет расходы в Google Sheets. Это удобно для быстрого ввода с телефона, но неудобно для нормального дашборда. Текущая таблица смешивает расходы, лимиты и накопления, поэтому модель становится неочевидной: пользователь вносит расходы, но показатели лимитов могут вести себя нелогично.

Новая модель должна оставить Google Sheets как временную базу данных и источник сырьевых операций, а логику расчетов и дашборды перенести в отдельное приложение, которое можно разместить на GitHub Pages.

## Главная продуктовая идея

Приложение должно отвечать на четыре вопроса:

1. На что я трачу деньги?
2. Сколько денег у меня реально свободно?
3. Сколько я могу потратить сегодня?
4. Если я продолжу тратить в таком темпе, хватит ли денег до следующей выплаты?

## Финансовая логика

Проект не должен жить по календарному месяцу. Денежный цикл строится вокруг выплат:

- зарплата — 15-го числа;
- аванс — 30-го числа;
- если дата выплаты попадает на выходной, выплата переносится на предыдущий рабочий день.

Базовая формула:

```text
Деньги сейчас
- обязательные платежи до следующей выплаты
- плановые накопления
= свободно до следующей выплаты

Свободно / дней до следующей выплаты = можно тратить сегодня
```

Накопления не должны лежать рядом с расходами. НЗ, отпуск и фонд машины — это не расходы, а цели распределения денег.

## Текущая кодовая база

Это стартовая база для Codex, не финальный продукт.

Состав:

- `index.html` — оболочка PWA;
- `src/app.js` — UI-логика, localStorage, добавление операций;
- `src/finance-model.js` — расчет финансового цикла, свободного остатка, дневного лимита и прогноза;
- `src/sheets-api.js` — заготовка клиента для Google Apps Script API;
- `apps-script/Code.gs` — заготовка Google Apps Script backend для чтения/записи Google Sheets;
- `data/sample-transactions.json` — seed-операции, извлеченные из текущей таблицы;
- `docs/DATA_MODEL.md` — описание модели данных.

## Как запустить локально

```bash
npm install
npm run start
```

После запуска открыть адрес, который покажет Vite.

## Как собрать

```bash
npm run build
```

Сборка появится в папке `dist`.

## Как разместить на GitHub Pages

1. Загрузить проект в GitHub-репозиторий.
2. Настроить сборку Vite через GitHub Actions или собрать локально и опубликовать `dist`.
3. В дальнейшем добавить переменную окружения `VITE_GOOGLE_SCRIPT_URL` со ссылкой на опубликованный Google Apps Script Web App.

## Google Sheets интеграция

Плановая схема:

```text
PWA на GitHub Pages
→ fetch к Google Apps Script Web App
→ Apps Script читает/пишет Google Sheets
→ PWA строит дашборды
```

Google Sheets должен хранить сырьевые таблицы:

- `Transactions`
- `Accounts`
- `Categories`
- `Obligations`
- `Goals`
- `IncomeSchedule`

На первом этапе можно поддержать только `Transactions`, а остальные настройки держать в `src/finance-model.js`.

## Задачи для Codex

1. Пересобрать Google Sheets-структуру под новую модель, сохранив только реальные транзакции из старой таблицы.
2. Доработать `apps-script/Code.gs`, чтобы он создавал недостающие листы и валидировал заголовки.
3. Добавить нормальные настройки счетов, обязательств и целей через UI.
4. Добавить экран редактирования операций.
5. Добавить фильтр по текущему финансовому циклу.
6. Добавить графики расходов по категориям без перегруза интерфейса.
7. Добавить режим offline-first: локальная запись → очередь синхронизации → отправка в Sheets при доступности сети.
8. Добавить PIN/локальную защиту приложения, если проект останется без полноценной авторизации.
9. Добавить импорт/экспорт JSON и CSV.
10. Подготовить GitHub Actions для деплоя на GitHub Pages.

## Важные требования к UX

- Ввод расхода должен занимать несколько секунд.
- Главный экран должен сразу показывать: свободно, можно сегодня, темп расходов, прогноз до выплаты.
- Не превращать продукт в бухгалтерию.
- Не плодить категории без необходимости.
- Не показывать пользователю сложные формулы на главном экране.
- Накопления отделять от расходов.
